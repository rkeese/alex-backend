# Alex Backend

**A modern, feature-rich club administration backend built with Go — because managing a club shouldn't feel like running an enterprise.**

Alex handles everything a club needs under one roof: members, finances, SEPA direct debits, documents, calendar events, board management, and more. It ships as a single binary backed by PostgreSQL, wrapped in Docker, and secured with JWT-based RBAC that goes far beyond basic auth.

---

## Why Alex?

Most club management tools are either bloated commercial products or bare-bones spreadsheets. Alex sits in the sweet spot — a clean REST API that covers the real-world complexity of German club administration (*Vereinsverwaltung*), from SEPA mandate tracking to SKR49 accounting charts, without dragging in a framework kitchen sink.

**What makes it interesting:**

- **Zero-framework Go** — Pure `net/http` with Go 1.22+ method routing. No Gin, no Echo, no magic. Just clean, readable handlers.
- **Type-safe SQL** — All database queries generated by [sqlc](https://sqlc.dev). No ORM, no runtime surprises.
- **German club finance built-in** — SEPA XML generation (pain.008.001.02), SKR49 account charts, bank booking imports from Sparkasse/Volksbank CSV formats, annual financial statements with PDF export.
- **Granular RBAC** — Five permission areas × three access levels × club-scoped roles. A treasurer sees finance data, a department head sees members — nobody sees what they shouldn't.
- **Production-hardened auth** — Bcrypt (cost 14), IP-based rate limiting, automatic account lockout after failed attempts, admin unlock, security logging.

## Feature Overview

### Members & Clubs
- Full member lifecycle: creation, import/export (CSV), invitation via email, statistics
- Milestone birthday lists (50, 60, 70, 80, 90, 100) and membership anniversary reports — with PDF export
- Hierarchical departments with parent-child relationships
- Board of directors management with automatic user provisioning

### Finance
- **Booking accounts** with German non-profit accounting classifications (ideeller Bereich, Vermögensverwaltung, Zweckbetrieb, wirtschaftlicher Geschäftsbetrieb)
- **Receipts & invoices** with VAT, line items, and receipt-to-booking transfer
- **SEPA direct debit** XML generation — grouped by creditor, supporting FRST/RCUR/OOFF/FNAL sequence types, delivered as a ZIP archive
- **Bank booking imports** — upload Sparkasse or Volksbank CSV files, review in a staging area, then commit
- **Annual financial statements** with per-account breakdowns and PDF generation
- **Donation receipt PDFs** for tax-deductible contributions

### Documents & Calendar
- Flexible document management with customizable categories (protocols, contracts, invoices, correspondence, …)
- Calendar events with month/year filtering and PDF export

### Security & Auth
- JWT authentication (HS256, 24h expiry) with role-embedded tokens
- Multi-level RBAC: `viewer` → `writer` → `manager` → `admin` → `sys_admin`
- Brute-force protection: 10 req/min per IP rate limit + 5-attempt account lockout (15 min)
- Proxy-aware (X-Forwarded-For, X-Real-IP)

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│  HTTP Request                                           │
├─────────────────────────────────────────────────────────┤
│  Rate Limiter (auth endpoints)                          │
│  ↓                                                      │
│  Auth Middleware (JWT validation)                        │
│  ↓                                                      │
│  Club Context Middleware (X-Club-ID / path param)       │
│  ↓                                                      │
│  Permission Check (RequirePermission / RequireSysAdmin) │
│  ↓                                                      │
│  Handler → Database (sqlc/pgx) → PostgreSQL             │
│         → PDF Generator (gofpdf)                        │
│         → SEPA XML Builder                              │
│         → Email Sender (SMTP)                           │
└─────────────────────────────────────────────────────────┘
```

### Project Structure

```
cmd/api/              → Main server entrypoint
internal/
  api/                → HTTP handlers, middleware, routing, rate limiting
  auth/               → JWT generation/validation, password hashing
  config/             → JSON-based configuration loader
  database/           → sqlc-generated queries and models
  email/              → SMTP sender for invitations & notifications
  pdf/                → PDF generators (birthdays, anniversaries, receipts, statements, events)
  sepa/               → SEPA pain.008.001.02 XML generation
migrations/           → 21 PostgreSQL migration scripts
queries/              → SQL source files for sqlc code generation
```

## Getting Started

### Prerequisites
- Go 1.25+
- Docker & Docker Compose
- PostgreSQL 15 (or use the Docker container)

### Run with Docker (recommended)
```bash
docker compose up -d --build
```
This spins up the database, backend API (port 8080), and frontend (port 8081).

### Local Development

Start only the database:
```bash
docker compose up -d db
```

Run the backend:
```bash
go run ./cmd/api/main.go
```

Reset the database (deletes all data):
```bash
docker compose down -v; docker compose up -d db
```

### Configuration

Configure via `config/config.json`:
```json
{
    "port": "8080",
    "database_url": "postgres://user:password@localhost:5432/alex_backend?sslmode=disable",
    "jwt_secret": "change_me_in_production"
}
```
Docker deployments use `config/config.docker.json`.

## Database

21 migration scripts evolve the schema from initial setup through RBAC, SEPA mandates, SKR49 accounting charts, invoice support, financial statements, document categories, and brute-force protection. Migrations are applied automatically on startup.

SQL queries are defined in `queries/*.sql` and compiled to type-safe Go code via sqlc — no string concatenation, no injection risks, no surprises.
